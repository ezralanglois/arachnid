file_processor

Framework for independent processing of files or groups of objects in serial, in parallel on a workstation or in parallel on a cluster using MPI.

The file processor module provides basic functionality to any program that processes a set of files or a set of object groups independently. It can distribute processing to multiple cores using the multiprocessing package as well as to multiple nodes on a cluster using mpi2py.

The target module must define the following functions:

process(filename, **extra)

Process a single file. During parallel processing this is invoked by a worker process.

Parameters:
  • filename – Input filename to process
  • extra – Unused keyword arguments (Options from the command line or config file, plus additional options)
Returns:

At minimum filename, however a tuple of additional arguments can be returned for processing by reduce_all()

The target module may include any of the following functions:

initialize(files, extra)

Initialize the input data before processing. During parallel processing this is invoked by all processes.

Parameters:
  • files – List of input files to process
  • extra – Dictionary of unused keyword arguments (Options from the command line or config file, plus additional options)
Returns:

None or a new list of files or objects to be processed by other routines
finalize(files, **extra)

Finalize data after processing. During parallel processing this is invoked by all processes.

Parameters:
  • files – List of input files to process (possibly modified by initialize())
  • extra – Unused keyword arguments (Options from the command line or config file, plus additional options)
reduce_all(filename, file_index=<int>, file_count=<int>, file_completed=<int>, **extra)

Reduce data from worker to root process. During parallel processing this is invoked by the root process. Note that only filename is a positional argument.

Parameters:
  • filename – Input filename that was processed. May be a tuple containing data resulting from process().
  • file_index – Current index in the list of input filenames
  • file_count – Number of input filenames
  • file_completed – Current count of finished files
  • extra – Unused keyword arguments (Options from the command line or config file, plus additional options)
Returns:

Only the filename that was processed
init_process(**extra)
Initialize the input data before processing. During parallel processing this is invoked by all processes only once before processing except the root.
Parameters:extra – Keyword arguments (Options from the command line or config file, plus additional options)
Returns:A dictionary of keywords to add or update.
init_root(files, extra)

Initialize the input data before processing (first init to be called). During parallel processing this is invoked only by the root process.

Note

This runs before dependency checking, so this is the place to generate groups. The first element of the group tuple must be an integer ID.

Parameters:
  • files – List of input files to process
  • extra – Dictionary of unused keyword arguments (Options from the command line or config file, plus additional options)
Returns:

None or a new list of files or objects to be processed by other routines

Each function also has access to the following keyword arguments:

  • finished: List of input files that have been processed and thus will not be processed this round
  • id_len: Maximum number of digits in the SPIDER ID

Parameters

The following parameters are added to a script using the file/group processing module in the program architecture.

--id-len <int>

Set the expected length of the document file ID

-w <int>, --work-count <int>

Set the number of micrographs to process in parallel (keep in mind memory and processor restrictions)

--force <bool>

Force the program to run from the start

--restart-test <bool>

Test if the program will restart

check_dependencies(files, infile_deps, outfile_deps, opt_changed, force=False, id_len=0, data_ext=None, restart_test=False, **extra)

Generate a subset of files required to process based on changes to input and existing output files. Note that this dependency checking is similar to the program make.

  1. Check if output files exist
  2. Check modification times of output against input
  3. Check if opt_changed flag was set to True
  4. Check if force flag was set to True
Parameters:
files : list

List of input files

infile_deps : list

List of input file dependencies

outfile_deps : list

List of output file dependencies

opt_changed : bool

If true, then options have changed; restart from beginning

force : bool

Force the program to restart from the beginning

id_len : int

Max length of SPIDER ID

data_ext : str

If the dependent file does not have an extension, add this extension

restart_test : bool

Test if program will restart

Unused extra keyword arguments

List of input filenames to process

finished : list

List of input filenames that satisfy requirements and will not be processed.

main
main(files, module, **extra)

Main driver for a generic file processor

The workflow proceeds as follows.

  1. For the root process, call init_root() and update any input files
  2. Test if a subset of files have already been processed and remove these
  3. Broadcast new set of files to be processed to all nodes (if necessary)
  4. Broadcast files already processed to all nodes (if necessary)
  5. ??
  6. Initialize data for each process including root
  7. If no files, finalize and return
  8. Initialize data for each worker process
  9. Process files
  10. Reduce data to root worker
  11. Finalize data on root worker
>>> from core.app import file_processor
>>> from arachnid.util import crop
>>> file_processor.main(['stack_01.spi'], crop)
Parameters:
files : list

List of filenames, tuple groups or lists of filenames

module : module

Main module containing entry points

extra : dict

Unused extra keyword arguments
